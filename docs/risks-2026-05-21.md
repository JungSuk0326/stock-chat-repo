## TL;DR — 즉시 보강 추천 3개

다음 순서로 처리하면 손실 위험이 가장 큰 부분부터 막을 수 있다.

### 1. R4 — 워커 토폴로지 결정 (지금 결정, 백엔드 코드 짜기 전)
- **별도 `worker` Docker 서비스 명시**를 CLAUDE.md 디렉토리 구조 + `docker-compose.yml`에 반영
- 이유: 워커를 API 프로세스 안에 두면 나중에 분리 비용이 큼. 처음부터 별도가 깔끔.

### 2. R2 — LLM 비용 hard cap (첫 Anthropic 호출 코드 짜기 전)
- `app/llm/budget.py` — Redis 카운터 기반 일/월 토큰 한도
- 이유: 무한루프 버그 한 번 = 수십만 원 청구. dev 모드는 1/10로 더 좁게.

### 3. R3 — Cloudflare Access (첫 외부 배포 전)
- Cloudflare Tunnel 위에 이메일/OAuth 인증 한 겹
- 이유: 단일 `AUTH_PASSWORD`로 외부 노출하면 brute force 한 번에 Anthropic 키까지 노출.

## CRITICAL — 실제 손실로 이어질 수 있음

### R1. 외부 데이터 소스 단일 의존, fallback 명시 없음
- **위치**: 한국 데이터 소스 전부 (`pykrx`, 네이버 모바일 API, DART, 종토방)
- **위협**: 네이버 모바일 API 한 번 막히면 실시간 시세 + 종토방 동시 손실. 두 개가 같은 도메인 변경에 동시에 깨질 수 있음.
- **현재 명시**: "비공식, 본인용이라 OK" — 끊겼을 때 동작은 없음
- **보완**:
  - `services/market/kr.py`에 primary + fallback 패턴. 실시간 실패 → pykrx EOD로 자동 강등
  - 종토방 끊기면 silent fail + LLM 컨텍스트에서 자동 skip (앱 자체는 계속 동작)
  - 헬스체크: 각 소스의 success rate 메트릭 → 24h 내 50% 미만이면 텔레그램 경고

### R2. LLM 비용 폭주 가능성, hard cap 없음
- **위치**: `services/llm.py:assemble_context()` 호출 빈도 + Anthropic 직접 호출
- **현재 명시**: 1500 토큰 목표, 캐시 TTL 60초
- **누락**: 일/월 토큰 cap, hard ceiling, 사용량 알림
- **위협 시나리오**: WebSocket으로 종목 빠르게 전환하면 분당 60+ 호출. 무한루프 버그 한 번 = $100+ 손실. opus-4-7은 토큰당 비싸므로 더 위험.
- **보완**:
  - `app/llm/budget.py` — Redis `INCR`로 일/월 토큰 누적 (atomic, race condition 없음)
  - hard ceiling 초과 시 즉시 `LLMBudgetExceeded` raise + 텔레그램 알림
  - dev 환경은 일 한도 1/10로 (실수 비용 차단)
  - 모든 Claude SDK 호출에서 `usage.input_tokens + usage.output_tokens`를 그대로 누적

### R3. 외부 노출 + 단일 비밀번호 = 공격면
- **위치**: Cloudflare Tunnel + `AUTH_PASSWORD` 환경변수 하나
- **위협**: 누군가 URL 알아내고 비밀번호 brute-force → Anthropic API key, DB 전부 노출. API key 노출되면 청구서 폭발.
- **현재 명시**: "본인용 단순 인증"
- **보완** (우선순위):
  1. Cloudflare Access 한 겹 추가 — 무료, 이메일/OAuth 기반. 본인용엔 가장 합리적
  2. `AUTH_PASSWORD` 기본값(`change-me`) 그대로면 앱 startup에서 거부
  3. 워커는 절대 외부 포트 X (API만 외부 노출)

### R4. 워커-API 토폴로지 미정
- **CLAUDE.md**: `APScheduler (배치/스케줄링)` / 어디서 뜨는지 불명확
- **시나리오 A** (같은 프로세스): `uvicorn --reload`로 워커 죽음, 다중 worker 띄우면 스케줄 중복 실행
- **시나리오 B** (별도 프로세스): `docker-compose.yml`에 worker 서비스 정의 필요한데 현재 없음
- **보완**: 별도 `worker` 컨테이너 권장
  - `services/worker` 추가, 같은 이미지 다른 entrypoint
  - 워커 singleton 보장은 Redis `SET NX EX` 락
  - API는 stateless 유지

### R5. 백업 전무
- **현재**: PostgreSQL 데이터 = 시놀로지 named volume `pgdata`만
- **위협**: NAS 디스크 손상 → 수년치 시세 히스토리 + candidate 스냅샷 영구 손실
- **보완**:
  - `workers/backup.py` — 일 1회 `pg_dump` → `/backup` 볼륨
  - 시놀로지 Hyper Backup으로 외부 저장소(B2/S3 무료 5GB)에 주 1회 오프사이트
  - TimescaleDB 압축이 켜져있다면 dump 사이즈 작음 — 거의 무료

## HIGH — 정확성 / 운영 안정성

### R6. LLM ticker 환각 안전장치 약함
- **현재**: "fundamentals 어댑터에서 상장폐지/거래정지 검증"
- **빠진 부분**: 검증 자체가 stale 캐시로 통과할 수 있음 (폐지된 종목인데 어제 캐시된 fundamentals 응답이 살아있음)
- **보완**:
  - 2단계 검증: (a) KRX 상장 종목 마스터 일치 + (b) 최근 5거래일 거래량 > 0
  - 둘 다 통과해야 `candidates` row 생성. 미통과는 `dismissed` + reason 로그

### R7. 종토방 약관/저작권 회색지대 더 강함
- **현재 명시**: "본문 DB 외부 노출 금지"
- **빠진 부분**: DB 내부 저장 자체도 위험. 네이버 약관에 무단 수집 금지 조항 + 한국 저작권법 사적이용(28조)은 "본인이 즉시 사용"까지만. DB 영구 보관은 사적이용 범위 초과 해석 여지.
- **보완**:
  - 본문 저장 안 함. 감성 라벨(positive/negative/neutral)과 시점만 저장
  - 분류 끝나면 본문은 메모리에서 즉시 폐기 (60초 미만 보유)
  - 어차피 LLM 컨텍스트엔 헤드라인/집계만 들어가니 기능 손실 없음

### R8. TimescaleDB 설계 디테일 부재
- **명시**: PostgreSQL 16 + TimescaleDB
- **누락**: hypertable 변환 시점, `chunk_time_interval`, 압축 정책
- **위험**: 1년 후 `prices` 테이블 10GB 넘어가면 일반 PostgreSQL 인덱스로는 쿼리 느림
- **보완**:
  - `prices` = hypertable, `chunk_time_interval = INTERVAL '7 days'` (KR + US 양쪽 적합)
  - 30일 이상 데이터는 `compress_chunk` 자동
  - 주의: Alembic은 hypertable native 지원 없음. 마이그레이션에서 raw SQL `SELECT create_hypertable(...)` 직접 작성 필요

### R9. 테스트 전략 미상세 + 외부 API 호출 가드 없음
- **명시**: `pytest + pytest-asyncio`
- **누락**: HTTP 모킹 도구 선택 (`respx`(httpx 전용) vs `vcrpy` vs `pytest-httpx`), 실제 외부 호출 차단 가드
- **위협**: `pytest` 그냥 돌리면 실제 DART API 호출 → key 노출 + rate limit 소진
- **보완**:
  - `respx` 추가 (httpx와 짝)
  - `conftest.py`에 autouse fixture로 외부 도메인 전체 차단. 명시적으로 모킹한 호출만 통과

### R10. 멀티마켓 추상화의 over-engineering 위험
- **CLAUDE.md 입장**: 처음부터 어댑터 3개(`market/`, `disclosure/`, `fundamentals/`) 분리
- **현실**: Phase 1엔 한국만. 추상 인터페이스 6개 미리 잡으면 미국 도래 시 인터페이스 자체가 한국 가정에 오염됐을 가능성
- **대안 입장** (dissenting): `market/`만 시장별 분기. `disclosure/`, `fundamentals/`는 일단 KR 전용으로 짜고 미국 올 때 그때 추상화
- **비용 비교**: 지금 추상화 = ~200줄 boilerplate × 3개 + 잘못 설계할 리스크. 나중에 추상화 = ~400줄 리팩토링 1회
- **권장**: 후자. CLAUDE.md 수정 권장
