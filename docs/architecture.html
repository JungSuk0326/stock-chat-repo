01 개요

제품의 한 줄, 그리고 무엇을 안 하는지.

핵심 가치

  • 실시간 시세 + 차트 + 공시 + 뉴스 + 기술 지표를 한 화면에 통합
  • 현재 보고 있는 종목의 컨텍스트를 자동으로 LLM에 주입해서 자연어 상담 가능
  • 가격/등락률/조건 기반 텔레그램 알림 + 채팅에서 자연어로 등록 가능 (LLM tool-use)
  • 스크리너 기반 종목 발굴 (기술적 + 재무적 양축) — 후보는 watchlist와 분리
  • 본인 1명 사용 — 단일 사용자 전제로 단순화, 멀티유저 골격만 유지

안 하는 것 스코프 외

  • 자동 매매 / 주문 실행
  • 회원가입·결제·이메일 인증
  • 종토방/뉴스 본문 노출
  • 다중 사용자 동시 사용

설계 원칙

  • 처음부터 멀티마켓 전제 (KR 외 US 등 확장 가능)
  • 본문은 어디에도 저장 X — 헤드라인만
  • 외부 API 실패해도 silent fail + 다음 tick 재시도
  • 도메인 어댑터 ABC로 시장별 분리

주요 정책

  • DB 시간은 UTC 저장, 표시만 KST
  • 종목 식별자 {EXCHANGE}:{SYMBOL} (예: KR:005930)
  • LLM 비용 hard cap (Redis 토큰 카운터)
  • 쓰기 도구는 사용자 확인 카드 거쳐서 실행

02 시장 확장 로드맵

코드/스키마는 처음부터 멀티마켓 전제로 설계. 한국 전용으로 박지 않음.

Phase 2 · 예정

미국 (NYSE / NASDAQ)

yfinance, SEC EDGAR, 영문 뉴스/Reddit 추가. 어댑터 ABC에 구현체만 추가.

Phase 3+ · 가능

일본·홍콩·유럽

필요 시. 같은 어댑터 패턴으로 확장 가능.

03 시스템 아키텍처

3 프로세스 · 2 데이터스토어 · 외부 API 7~8종.

데이터 수집 / 저장 LLM 사용자 인터페이스 운영 / 인프라

Browser Next.js 16 · React 19

  • 차트 (lightweight-charts) + 지표 토글 (SMA/EMA/Bollinger/RSI)
  • Watchlist 사이드바 + 종목 검색 모달
  • 공시 / 뉴스 탭 (MarketFeedPanel)
  • 알림 패널 (수동 + AI 자연어 둘 다)
  • ChatPanel — 세션 영속화 + tool-use 확인 카드
  • WebSocket으로 실시간 tick 수신

Backend (FastAPI) app/main.py

  • REST: /prices /instruments /watchlist /chat /chat/sessions /chat/tool-confirm /llm/models /disclosures /news /alerts /health
  • WebSocket: /ws/prices/{ex}/{sym} (Redis Pub/Sub fan-out)
  • LLM 추상화 (Anthropic + Gemini) + 도구 dispatcher + multi-turn loop
  • 종목 컨텍스트 어셈블러 (assemble_context)
  • watchlist 추가 시 즉시 백필 (가격 동기 / 공시·뉴스 비동기)

Worker (APScheduler) app/workers/runner.py

  • 2초 · 종목별 price_poller_* (네이버 시세 → Redis + DB)
  • 30초 · watchlist_sync (reconcile, 동적 잡 추가/제거)
  • 1분 · disclosure_poll · alert_runner
  • 5분 · news_poll
  • 일일 cron 6개 · 자세한 건 §06
  • Heartbeat → Redis → /health.workers에 stale 자동 노출

저장소 Docker compose

  • PostgreSQL 16 + TimescaleDB — prices hypertable (7d chunk, 30d 압축)
  • Redis 7 — 가격 캐시 (price:{EX}:{SYM}), pub/sub (ticks.{EX}.{SYM}), LLM 토큰 카운터, 워커 heartbeat
  • 매일 03:30 KST pg_dump → ./backups/ (오프사이트는 호스트 Hyper Backup)

외부 API

  • 네이버 모바일 — 실시간 시세 (2s 폴링) + 종목별 뉴스 (5min)
  • pykrx — EOD 일봉 (16:00 KST)
  • FinanceDataReader — KRX 종목 마스터 (06:00 KST)
  • DART OpenAPI — 공시 + corp_code (1min poll + 05:30 daily)
  • yfinance — KR fundamentals (PER/PBR/marketCap, .KS/.KQ)
  • Anthropic Claude (Opus 4.7, Haiku 4.5) + Google Gemini (2.5 Pro, 2.5 Flash)
  • Telegram Bot API — 알림 발송 채널

04 도메인별 데이터 흐름

기능별로 어떻게 데이터가 움직이는지.

① 실시간 시세 (2초 폴링)

[네이버 모바일 API]
       │ HTTP 2초마다
       ▼
[Worker: price_poller]
       ├─→ Redis SET  price:{EX}:{SYM}  TTL 60s     (현재가 캐시)
       ├─→ Redis PUB  ticks.{EX}.{SYM}              (실시간 fan-out)
       └─→ 분 경계 DB INSERT prices(interval='1m')  (히스토리 영속)

                       │  Redis SUBSCRIBE
                       ▼
[Backend WebSocket /ws/prices/{ex}/{sym}]
                       │ 서버 → 브라우저 push
                       ▼
[Browser: chart.update(tick) + 지표 실시간 재계산]

② 공시 (DART, 1분 폴링)

[DART OpenAPI] ──→ Worker: disclosure_poll (1min)
                       │ watchlist 전체 × [today-1, today] KST
                       │ ON CONFLICT(source, source_id) DO NOTHING
                       ▼
                   [DB: disclosures]
                       ├─→ /disclosures REST → 프론트 공시 탭
                       └─→ assemble_context → LLM 컨텍스트 (최근 14일/20건)
신규 watchlist 종목: 백필 fire-and-forget (6개월)

③ 뉴스 (네이버 금융, 5분 폴링)

[네이버 m.stock] ──→ Worker: news_poll (5min)
                       │ watchlist 전체 × pageSize=30
                       │ UNIQUE(source, source_id) 멱등
                       │ 본문 X — 헤드라인 + publisher + URL만
                       ▼
                   [DB: news_items]
                       ├─→ /news REST → 프론트 뉴스 탭
                       └─→ assemble_context → LLM 컨텍스트 (최근 24시간/15건)
신규 watchlist 종목: 백필 fire-and-forget (50건)

④ 투자자 수급 (Top8, 일일 16:30 KST)

[네이버 m.stock /api/stock/{symbol}/trend] ──→ Worker: investor_flow_sync_daily
                       │ watchlist 전체, pageSize=30 (~30거래일)
                       │ UNIQUE(instrument_id, trade_date) 멱등
                       │ 신규 watchlist 종목 backfill 60일
                       ▼
                   [DB: investor_flows]
                       ├─→ /investor-flows REST → 프론트 [수급] 탭 (테이블)
                       └─→ assemble_context → LLM 컨텍스트 priority #4
                            "외국인: 순매수 −17.71M주 (매수 0일/매도 5일)"
                            5거래일 집계 + 매수/매도일 카운트 (스토리가 드러남)

⑤ 알림 (1분 평가 + 텔레그램)

[사용자]
   │ 수동 UI 폼     ↘
   │                 ╲
   │ ChatPanel        → app/services/alert_rules.create_rule
   │ "260k 알림"     ╱      │
   │ ↓ LLM tool-use ↗       ▼
   │ pending_actions     [DB: alert_rules] (UNIQUE 멱등)
   │ → 확인 카드 → /chat/tool-confirm
   │
   │  Worker: alert_runner (1min)
   ▼
   ┌────────────────────────────────────────┐
   │ for each enabled rule:                  │
   │   - cooldown 검사 (last_triggered + N분) │
   │   - market_hours_only면 장 시간 체크    │
   │   - 가격 평가 (Redis 캐시 우선)         │
   │   - 통과 시: TelegramChannel.send()     │
   │            + alert_events INSERT        │
   │            + last_triggered_at = now()  │
   └────────────────────────────────────────┘
        ↓
   [Telegram 봇 → 사용자 폰]

⑥ LLM 채팅 (multi-turn tool-use)

[사용자 질문]
   │
   ▼
[POST /chat]
   │
   ├─ 1. assemble_context(symbol)
   │     └─ 시세 + 추세 + 지표 + 공시(14d/20) + 뉴스(24h/15) + 수급(5거래일 집계)
   │        총 ~1500 토큰 목표, 현재 ~460 토큰 사용
   │
   ├─ 2. system prompt + 컨텍스트 + tools + history → LLM 호출
   │     (provider/model은 ChatPanel 드롭다운에서 선택, 매 호출마다 전달)
   │
   ├─ 3. tool_calls 있나?
   │     ├─ 읽기 (list_alerts) → 즉시 실행 → tool_result 다시 LLM에 → 최종 답변
   │     └─ 쓰기 (create_alert_rule / delete_alert)
   │            └─ pending_actions로 응답 (실행 X)
   │
   ├─ 4. chat_messages INSERT (user + assistant 영속화)
   │
   └─ 5. 첫 메시지면 background에서 자동 타이틀 생성 (Gemini Flash)
        │
        ▼
[Browser ChatPanel]
   ├─ 답변 메시지 표시
   ├─ pending_actions 있으면 ActionCard 렌더 ([확인][취소])
   │     └─ [확인] → POST /chat/tool-confirm → 실제 INSERT → CustomEvent로 AlertsPanel 즉시 갱신
   └─ 세션 드롭다운 / 새 대화 / 삭제

⑦ 발굴 — 스크리너 (Top7, 일일 17:30 KST)

[Screener 정의 (DB)]
   universe: {market: "KOSPI"}
   criteria: [
     {type: "technical:rsi_below", value: 30},
     {type: "fundamental:per_below", value: 10}

[Worker: screener_run_daily 17:30 KST]
        │

        ├─ 2. 기술 필터 (DB 일괄 쿼리, 종목별 RSI/SMA/52w/거래량 계산)

        ├─ 3. 재무 필터 (yfinance 캐시 우선, MAX 200 종목 호출 cap)
        │      → 최종 후보
        └─ 4. candidates INSERT (status="new")
              · 기존 status=dismissed면 건너뜀 (한 번 거절한 건 재발견 X)
              · 기존 snoozed + snoozed_until 미래면 건너뜀
              · 그 외 status=new + reason 갱신

[Frontend 발굴 UI] — Phase D 미구현
   후보 리스트 + [승급][스누즈][폐기] 액션

05 DB 스키마

11개 테이블. 모두 멀티마켓 전제로 exchange 컬럼 포함.

users 멀티유저 골격

현재 row 1개 (id=1, "owner"). 인증은 AUTH_PASSWORD 환경변수. R3에서 Cloudflare Access로 교체 예정.

instruments 종목 마스터

exchange / symbol / country / currency / market / isin / name. UNIQUE(exchange, symbol). 매일 06:00 KST FDR로 KOSPI/KOSDAQ ~2,649개 UPSERT.

watchlist 관심 종목

instrument_id UNIQUE + position. 추가 시 가격/공시/뉴스 즉시 백필 fire-and-forget.

prices hypertable

TimescaleDB 7d chunk, 30d 이후 자동 압축. interval='1d'/'1m'. 1년치 일봉 + 분봉 누적.

corp_codes

DART corp_code(8자리) ↔ stock_code(6자리) 매핑. 매일 05:30 KST 갱신, 상장사 ~3,900개. 추후 SEC CIK도 같은 테이블.

disclosures

instrument_id FK + UNIQUE(source, source_id). title/filed_at(UTC)/raw_url. 본문 X. 매분 폴링.

news_items

네이버 금융. UNIQUE(source, source_id). title/published_at/url/publisher. 본문 X. 5분 폴링.

investor_flows Top8

일별 외국인/기관/개인 순매수 + 외국인 보유율. UNIQUE(instrument_id, trade_date). 일일 16:30 KST cron.

fundamentals_snapshot 캐시

yfinance 결과. UNIQUE(instrument_id). per/forward_per/pbr/dividend_yield/market_cap/beta/sector. TTL 24h, screener 실행 시 lazy refresh.

chat_sessions

user_id + instrument_id + title(자동 생성). updated_at으로 최근순 정렬.

chat_messages

session_id FK + role/content + model/input_tokens/output_tokens. CASCADE delete.

alert_rules

user_id + instrument_id + condition_type + threshold + cooldown_minutes + last_triggered_at. condition_type 4종.

alert_events

rule_id FK + fired_at + triggered_value + channel + delivery_status. 발화 이력/디버깅.

screeners Top7

universe JSONB + criteria JSONB + enabled. 일일 17:30 cron으로 자동 실행.

candidates Top7

UNIQUE(user, instrument, source). status: new/snoozed/promoted/dismissed. 스크리너 재실행 시 dismissed는 재발견 X.

시간대 규칙: 모든 timestamptz는 UTC로 저장. 표시할 때만 KST(Asia/Seoul)로 변환. 시장 시간 비교는 거래소 로컬 시간 기준.

06 워커 잡 시간표

APScheduler 단일 worker 프로세스 안에서 모두 돌아감. /health.workers에서 잡별 상태 확인.

잡 이름주기역할상태 카테고리
price_poller_{EX}_{SYM} 2초 interval 종목별 실시간 시세 (Naver) → Redis + 분봉 적재 동적
watchlist_sync 30초 interval watchlist DB 변경 reconcile, 동적 잡 add/remove interval
disclosure_poll 1분 interval DART 공시 watchlist 전체 [today-1, today] KST interval
alert_runner 1분 interval 활성 룰 평가 + cooldown + Telegram/Log channel 발화 interval
news_poll 5분 interval 네이버 금융 뉴스 watchlist 전체 sweep interval
investor_flow_sync_daily 16:30 KST cron Naver trend API로 외국인/기관/개인 순매수 + 외국인 보유율, 30일 윈도 daily
backup_daily 03:30 KST cron pg_dump | gzip → BACKUP_DIR. 14일 보존, 호스트가 오프사이트 (Hyper Backup) daily
dart_corp_code_sync_daily 05:30 KST cron DART corp_code ZIP/XML → corp_codes 테이블 UPSERT (~3,900개) daily
instruments_sync_daily 06:00 KST cron FDR로 KOSPI/KOSDAQ 전 종목 마스터 갱신 daily
eod_sync_daily 16:00 KST cron KOSPI 종가 후 30분, pykrx로 watchlist 최근 7일 일봉 갱신 daily
fundamentals_refresh_daily 17:00 KST cron yfinance로 watchlist fundamentals 캐시 keep-warm daily
screener_run_daily 17:30 KST cron 활성 스크리너 전부 평가 → candidates 적재 daily
R1 헬스 메트릭: 각 잡은 with_heartbeat 래퍼로 감싸져 Redis에 last_ok_ts 기록. interval 잡 5분 / daily 잡 25시간 임계 넘으면 /health가 503 반환 → 워커가 silent하게 죽으면 즉시 알아챔.

07 기술 스택

Backend

  • Python 3.12 + uv (deps)
  • FastAPI · async
  • SQLAlchemy 2.0 async + asyncpg
  • Alembic (멀티마켓 마이그레이션, env.py에 TimescaleDB 인덱스 필터)
  • APScheduler (배치/스케줄, AsyncIO)
  • httpx (외부 API), structlog
  • Anthropic SDK + google-genai SDK
  • yfinance (Phase 1 KR fundamentals + 환율, Phase 2 US 본격)

Frontend

  • Next.js 16 (App Router, Turbopack)
  • React 19 + TypeScript strict
  • Tailwind CSS v4
  • Lightweight Charts v5 (panes API로 RSI 별도 페인)
  • Browser WebSocket (커스텀 재연결 backoff)

저장소 / 인프라

  • PostgreSQL 16 + TimescaleDB (compose, healthcheck 포함)
  • Redis 7 (cache + pub/sub + 토큰 카운터 + heartbeat)
  • Docker Compose (postgres / redis / backend / worker)
  • 워커 이미지에 postgresql-client-16 (pgdg apt repo)

외부 / 미래

  • 시놀로지 NAS 배포 예정
  • Cloudflare Tunnel + Access 인증 (R3)
  • Gitea (kovis.synology.me) — Git 저장소
  • 오프사이트: B2/S3 무료 티어 + Hyper Backup

09 주요 설계 결정

왜 그렇게 만들었는지 — 미래의 본인/리뷰어 대상.

워커 / API 분리 R4

같은 Docker 이미지, command만 다르게. API 다중 인스턴스 띄워도 워커 중복 실행 X. 워커 OOM이 API 안 죽임. 워커는 외부 포트 노출 0.

LLM 비용 cap R2

Redis INCRBY 원자 카운터로 daily/monthly 토큰 누적. provider 통합 한도. 초과 시 호출 전 LLMBudgetExceeded raise → 429. 무한 루프 버그로 청구서 폭발 방지.

본문 저장 절대 X

공시/뉴스/종토방 모두 헤드라인+메타만. 약관/저작권 회색지대 + 토큰 예산 + DB 사이즈 셋 다 해결. LLM 컨텍스트도 헤드라인까지만.

멀티유저 골격만 유지

지금 1명만 쓰지만 users 테이블 + user_id FK는 새 테이블엔 다 박음. 인증 자체는 AUTH_PASSWORD 그대로, get_current_user_id dependency 1곳만 교체하면 multi-user 가능.

LLM 쓰기 도구는 확인 카드

create_alert_rule / delete_alert 호출 시 즉시 DB INSERT 안 함. pending_actions로 UI에 카드 띄우고 사용자가 [확인] 누를 때만 실행. 환각 안전망. 읽기 도구(list_alerts)는 즉시 실행 + LLM에 결과 다시 던짐.

fundamentals: pykrx 죽음 → yfinance

pykrx 1.2.8의 bulk fundamental 엔드포인트가 KRX upstream 변경으로 깨짐. yfinance가 KR도 .KS/.KQ로 지원 — Phase 2 미국용으로 어차피 들어올 예정이라 위치도 자연스러움.

스크리너 평가 순서 최적화

universe → 기술 필터(DB만, 빠름) → 통과 종목만 fundamental(yfinance, 느림). 기술적 사전 필터로 universe 1/10~1/100로 줄인 다음 외부 호출. 순수 기술 스크리너면 yfinance 0회.

실시간 = WebSocket + Redis Pub/Sub

워커가 2초마다 폴링 → Redis pub. Backend WebSocket이 sub → 브라우저로 fan-out. 다중 brower 클라이언트가 같은 종목 봐도 한 종목당 1개 폴링.
